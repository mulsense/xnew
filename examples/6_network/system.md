# 6_network — 仕組み解説

このサンプルが「どう動いているか」を、図とともになるべくやさしく説明します。
使い方は [manual.md](manual.md) を参照。

---

## 1. 一言でいうと

> **サーバーが正しい位置を計算し、ブラウザはそれを描くだけ**。
> そして **ルーム（部屋）ごとに別プロセス** がゲームを動かし、入口のポートは **1 つ**。

「ブラウザは入力を送る／結果を描く」「サーバーが本当の状態を持つ」という分担を
**サーバー権威モデル**と呼びます。チート耐性が高く、全員の画面が一致しやすいのが利点です。

---

## 2. 全体像

```
   ブラウザ                              サーバー（Node / 複数プロセス）
 ┌───────────┐                   ┌──────────────────────────────────────────┐
 │  画面(xnew) │                   │            master プロセス (:3000)          │
 │           │   ① 接続           │   - 唯一ポートを開く                         │
 │ lobbySocket├──────────────────▶│   - 接続を「行き先」で振り分けるだけ          │
 │           │                   │     （フレームは中継しない）                  │
 │ gameSocket ├───────┐           └───┬───────────┬───────────┬───────────┬────┘
 └───────────┘       │ ② 接続      ハンドオフ    │           │           │
                     │  (room=r2)      ▼          ▼           ▼           ▼
                     │            ┌────────┐ ┌────────┐ ┌────────┐ ┌────────┐ ┌────────┐
                     └───────────▶│ lobby  │ │ room r1│ │ room r2│ │ room r3│ │ room r4│
                                  │ worker │ │ worker │ │ worker │ │ worker │ │ worker │
                                  └────────┘ └────────┘ └───┬────┘ └────────┘ └────────┘
                                  静的配信        各ルーム = 1 プロセス
                                  ロビー一覧      xnew でゲームループ（Arena/Player）
                                                     │
                                          ③ 30Hz で state を直接配信
                                                     ▼
                                            （その部屋のブラウザへ）
```

- **master**：ポート :3000 を開く唯一のプロセス。来た接続を「どのワーカーへ渡すか」だけ決めて、**接続そのもの（OS のソケット）をワーカーに引き渡す**。以降のやり取りには関与しない。
- **lobby worker**：HTML/JS/CSS の配信と、ロビー（ルーム一覧・人数）を担当。
- **room worker（×4）**：1 つのルーム専用。そのルームのプレイヤーの socket を直接持ち、xnew でゲームを回し、`state` を直接配信する。

---

## 3. サーバー権威モデル（位置の流れ）

ブラウザは「押している方向」だけを送り、位置計算はしません。サーバーが毎フレーム位置を更新し、
一定間隔で全員に配る。届いた位置をブラウザはそのまま描きます。

```
[ブラウザ]  キー入力 → 方向ベクトル {x,y} ──(input)──▶ [room worker]
                                                        │ 毎フレーム位置を計算
                                                        │ (サーバーが本当の位置を持つ)
[ブラウザ]  canvas に描画 ◀──(state: 全員の位置)── 30Hz ┘
```

ポイント：
- 入力は「変わったときだけ」送る（押しっぱなしは 1 回送れば十分。サーバーが次の入力まで動かし続ける）。
- 位置は整数に丸めて配信（帯域節約。表示には十分）。

---

## 4. いちばんの工夫：単一ポートでルーム毎プロセス

「ポートは 1 つ」「ルームは別プロセス」「毎フレームを中継するハブは作らない」を
**同時に**満たすための仕組みが **コネクション・ハンドオフ** です。

### 何が難しいのか

- ポートを 1 つにすると、普通は 1 プロセスが全員の通信を捌く＝そこがボトルネック。
- かといって state を作るプロセス（ルーム）と socket を持つプロセスが別だと、
  その間で「30Hz × 部屋数」の転送（＝中継）が必ず発生してしまう。

### どう解いたか

master は接続が来た瞬間に **最初の 1 行（HTTP ハンドシェイク）だけ** を覗いて行き先を決め、
**OS のソケットをそのワーカーへ渡してしまう**。渡した後は master は通信路から外れ、
ワーカーがブラウザと直接やり取りします。

```
ブラウザが /socket.io/?...&room=r2 で接続
        │
        ▼
   master が最初のチャンクを見る → "room=r2" を発見
        │
        │  worker.send({...}, socket)   ← ソケットを r2 ワーカーへ手渡し
        ▼
   room r2 worker がこの接続を所有
        │
        ▼
   以降、入力も state も「ブラウザ ⇄ r2 worker」で直接（master を通らない）
```

- ルーム判別のキーは接続 URL の `?room=r2`。**1 接続につき 1 回読むだけ**（フレーム毎ではない）。
- この「接続を別プロセスへ手渡す」技術は socket.io 公式の `@socket.io/sticky` を利用。
  ただし標準は session 単位の振り分けなので、**room 単位の振り分けだけ master 側で自前実装**している。

### なぜ「中継なし」と言えるか

| 処理 | 頻度 | 担当 |
| --- | --- | --- |
| 接続の振り分け | 接続ごとに 1 回 | master |
| 入力の受信 / 位置計算 / state 配信 | 毎フレーム | 各 room worker（クライアント直結）|
| ルーム人数の通知 | 参加/退出のときだけ | room → master → lobby |

毎フレーム発生する処理（フレーム × 部屋数）は **すべて各ルームワーカー内で完結** し、
master を経由しません。master の仕事は「接続ごとに 1 回」と「たまの人数中継」だけです。

---

## 5. なぜ socket.io の room 機能を使っていないか

各 room worker には、master が **そのルームの接続だけ** を振り分けます。
つまり 1 プロセス内のクライアントは必ず同じルーム。だから配信は `io.emit`（全員へ）で十分で、
socket.io の room（部屋分け）機能は要りません。「プロセス＝部屋」になっているからです。

---

## 6. xnew でのゲームループ（room worker の中）

ゲームロジックは xnew のコンポーネントで組んでいます。

```
Arena（部屋の管理＋配信）
 ├─ Player A   ← 参加時に xnew(Player) で動的に追加
 ├─ Player B
 └─ Player C   ← 退出時に finalize で破棄
```

- **Player**：自分の入力ベクトルに従い、自分の `update` で毎フレーム自分の位置を計算する。
- **Arena**：`join` で子 Player を生成、`leave` で破棄、`input` を該当 Player へ中継。
  そして自分の `update` で 30Hz に間引いて全 Player の位置を集めて配信する。

xnew の更新順は **子 → 親**。なので 1 フレームの中で

```
（全 Player が動く） → （Arena が配信する）
```

の順になり、配信される位置は常に最新です。

> サーバーもブラウザも、毎フレームの処理は同じ xnew の `update` / `render` イベントで回しています
> （サーバー = 位置計算、ブラウザ = 描画）。

## 7. なぜ接続が 2 本あるのか

ブラウザは socket を 2 本張ります（どちらも同じ :3000）。

| 接続 | 行き先 | 役割 |
| --- | --- | --- |
| `lobbySocket`（room 無し） | lobby worker | ルーム一覧・人数の受信 |
| `gameSocket`（`?room=rX`） | その room worker | 入力送信・位置受信 |

ロビーとゲームは別プロセスが担当するので、接続も分かれます。
ルームを変えると `gameSocket` を張り直し、別の room worker に着地します（＝接続先プロセスが替わる）。

---

## 8. 通しの流れ（参加〜移動）

```
1. ページを開く            ブラウザ ──GET /──▶ master ──(振り分け)──▶ lobby worker（HTML 等を返す）
2. 名前入力 → ロビー        lobbySocket ──▶ lobby worker ──(lobby:rooms)──▶ ルーム一覧を表示
3. ルーム r2 を選ぶ         gameSocket(room=r2) ──▶ master ──(ハンドオフ)──▶ room r2 worker
4. welcome 受信            room r2 ──(welcome: id/field/pid)──▶ ブラウザ → join 送信
5. プレイ                  キー ──(input {x,y})──▶ room r2（位置計算） ──(state 30Hz)──▶ 全員へ描画
6. 退出（ロビーへ）          gameSocket を閉じる → room r2 が Player を破棄 → 人数を lobby へ通知
```

ブラウザ側の画面遷移（xnew の Scene）：

```
JoinScene（名前）── unit.change ──▶ LobbyScene（ルーム選択）── unit.change ──▶ GameScene（プレイ）
```

## 9. イベント早見

| 方向 | event | payload | 意味 |
| --- | --- | --- | --- |
| C→S(lobby) | `lobby:enter` | — | ルーム一覧をください |
| S→C(lobby) | `lobby:rooms` | `{rooms:[{id,name,memberCount}]}` | ルーム一覧・人数 |
| S→C(room) | `welcome` | `{id, field, roomId, pid}` | 自分の id・フィールド・担当プロセス |
| C→S(room) | `join` | `{name}` | 参加します |
| C→S(room) | `input` | `{x, y}` | 方向（各軸 -1/0/+1） |
| S→C(room) | `state` | `{players:[{id,name,color,x,y}]}` | 全員の位置（30Hz）|

---

## 10. まとめ

- **サーバー権威**：位置はサーバーが計算、ブラウザは送る／描くだけ。
- **ルーム＝プロセス**：各ルームは専用プロセスで、そのルームの socket を直接持つ。
- **単一ポート＆中継なし**：master は接続を 1 回振り分けるだけ。フレームの通信は各ルームが直接担う。
- **xnew で統一**：サーバー（位置計算）もブラウザ（描画）も、同じ xnew の仕組みでループを回す。
